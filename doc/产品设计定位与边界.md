# 产品设计定位与边界

> 本文档记录 norn 的产品定位、设计取向、以及"做什么 / 不做什么 + 为什么不做"。
> 目的是在后续开发中保持产品边界不被功能膨胀破坏，所有功能与 UI 决策都应回到本文档对齐。
>
> 与现有《轻量代码编辑器与 Git 管理工具需求文档》的关系：那份是功能规格清单，本文档是**定位与取舍的总纲**。两者冲突时，以本文档的定位为准。

---

## 1. 一句话定位

> **norn 是一个原生、优雅的「改动查看 + 确认 + 提交」工具，附带轻量编辑能力。**

它不是编辑器，不是 IDE，也不是专业 Git 客户端。它是在"AI 已经把代码写完/改完"的前提下，帮用户**快速看清改了什么、顺手微调、然后提交**的工具。

---

## 2. 核心使用场景

主导动词是 **看 → 确认 → 提交**，"编辑"只是看的时候顺手点一下。

典型流程：

```
打开项目（多窗口 / 快速切换）
  → 看当前项目是什么样、AI 改了哪些、哪里没改、改了什么
  → 顺手改两个配置项或几行代码
  → 写一句提交说明，提交
  → 看到「刚提交了什么」+ 当前分支位置
```

关键场景特征：

- **AI 协作下的 review**：AI（codex / claude 等）改完代码后，用户需要快速 diff 扫查"哪改了、哪没改、改了啥"，确认无误再提交。**这是打开率最高的屏幕，是产品的核心价值。**
- **多项目 / 多窗口并行**：用户会同时开多个窗口、频繁切换，要求轻、快、状态一眼可知，而不是坐下来做一小时 Git 操作。
- **浅编辑**：编辑的多是配置项、少量代码，不需要编辑器智能。

---

## 3. 我们是什么 / 不是什么

| 我们是 | 我们不是 |
| --- | --- |
| 原生、优雅、轻量的改动查看与提交工具 | 编辑器（Zed / Cursor / VS Code 的竞品） |
| 围绕"看 diff + 提交"组织的工作流 | 完整 IDE |
| AI 写完代码后的 review / commit 收尾工具 | 专业 Git 管理平台（GitKraken / Fork / Tower / Sourcetree） |
| 调用外部 AI（codex / claude）的优雅外壳 | AI agent 编排平台 |

---

## 4. 为什么单独做，而不是用 Zed / Cursor

Zed / Cursor / VS Code 是方向上的灵感来源，但**不适合直接用**，原因不是它们不好，而是**它们的重量全压在我们不需要的那一侧**：

- 它们的核心资产是**编辑智能**——LSP、语法树、补全、重构、项目索引。这些我们都不需要（写代码的活儿 AI 干了）。
- 我们需要做到极致的那一侧——**原生质感、轻、快、多窗口、提交闭环**——它们反而不极致（普遍有 Electron 味）。

所以它们更多还是"带 Git 的编辑器"，而 norn 是"**带轻编辑的、原生优雅的改动查看与提交器**"，是不同品类。

---

## 5. 参考对象（换一批，别再看编辑器）

真正的对标是"**原生、优雅、聚焦做一件技术小事、多窗口**"的工具：

| 参考 | 学什么 |
| --- | --- |
| **TablePlus** | 结构骨架：原生、轻、多窗口、给技术用户做一件聚焦的事。学它的"原生质感 + 克制 + 快" |
| **Kaleidoscope**（macOS diff 工具） | diff / review 体验天花板：原生、漂亮、好读。我们的 diff 直接对标它 |
| **Tower** | 原生 Git 的精致度（**只学质感，不学功能广度**，它是专业工具） |
| **Raycast** | "快、键盘、优雅的命令入口"范式。将来接 codex / claude 的入口就该是这种：一个唤起、给结果、用户确认 |
| **Linear / Things 3** | 现代 UI 的克制与打磨基准 |
| **Files / Windows Terminal**（Windows 侧） | Fluent / Mica / WinUI 的原生现代感 |

> 反面参考：GitKraken / Fork / Sublime Merge 这类专业 Git 工具**不要照抄**——它们是项目中心、重型的，跟我们的定位相反。

## 6. 设计预算分配

既然编辑是浅的、智能交给 AI，力气要花在差异化上。

### 6.1 重投入（这是差异化，决定成败）

1. **原生 + 现代的平台质感**——"优雅"的真正载体，也是别人（Electron 系）做不好、而我们用 Tauri 原生壳有先天优势的地方。要吃干榨净：
   - macOS：vibrancy / 材质、原生标题栏与红绿灯、SF Pro、跟随系统强调色、明暗主题、尊重 reduce-motion。
   - Windows：Mica、Fluent 间距、Segoe UI Variable、原生标题栏。
2. **diff / review 体验**——"AI 改了哪、改了啥、哪没改"逐项扫查，做到 Kaleidoscope 级的清晰好读。
3. **提交闭环 + glanceable 状态**——写一句、提交、看到"提交了什么 + 当前分支位置"；多窗口切换时状态栏一眼可知该项目 Git 状态。
4. **快**——冷启动、打开项目、切窗口都要快，让人愿意同时开很多个。

### 6.2 刻意不做（保持轻，这是我们对 Zed / Cursor 的优势来源）

| 不做 | 不做的目的 |
| --- | --- |
| LSP、智能补全、语义跳转、重构、自动 import | 这些是编辑器的重量来源，做了就变成低配 IDE，丢掉"轻快"这一核心优势 |
| 项目级语义索引、查找引用、精确跳转定义 | 同上；且与"看 + 提交"的核心动作无关，属于编辑器智能范畴 |
| 重型语法树驱动的高亮 | 高亮只需"够读"，轻量着色即可；大文件直接降级纯文本，保证打开速度 |
| 内置 Git 引擎（libgit2 / JGit）、专业 Git 操作（rebase / cherry-pick / reset / force push 等） | 我们不是专业 Git 平台；直接调用系统 git 即可，复用用户已有配置 / 认证 / hooks，降低复杂度 |
| 全屏大型提交关系图（GitKraken 式） | 与"Git 次要、不占常驻空间"的定位冲突；用户只需提交后确认级别的轻量分支视图 |
| AI agent 编排、多步自动化 | 我们只做"一个唤起 codex / claude 的入口"，结果丢进 diff 给用户 review，剩下交给用户；编排会让产品失焦、变重 |
| 调试器、插件系统（第一阶段） | 超出"看 + 改两下 + 提交"的边界，是 IDE 范畴 |

## 7. 信息架构与交互骨架

### 7.1 整体布局

保持三栏，**不推翻**。Git 是次要的、靠后的、glanceable 的。

```
┌─ 标题栏（原生）── 分支 main ↑2 ──────────────────────────┐
├──────────┬──────────────────────────────┬────────────────┤
│ 文件树    │  内容区（读 / 轻编辑 / Diff）  │  Git（变更/提交）│
│（最高优先）│  平时=文件，review 时=diff     │  次要、靠后      │
├──────────┴──────────────────────────────┴────────────────┤
│ 状态栏：路径 · 行列 · 编码 · ◑main↑2 · 3 改动（glanceable）│
└────────────────────────────────────────────────────────────┘
```

层级理由：

- **左侧文件树优先级最高**——Git 依附于项目，"项目/文件"是一切的前提。
- **中央内容区**不是"编辑器"，而是"一个会看 diff 的优雅阅读 / 微调区"：平时显示文件，review 时显示 diff，提交后给轻量结果。编辑与 Git 都围着"内容/改动的查看"转。
- **右侧 Git** 次要、靠后，承载 review → 提交 → 确认 的收尾工作流。

### 7.2 右侧 Git：一条随工作流变化的"轨道"，而非静态面板

按操作流分阶段：

- **Review 态（默认）**：改动文件列表，清楚区分"已改 / 未改 / 新增 / 删除"。
- **Commit 态**：写 message → 提交。
- **已提交态**：显示"刚提交了什么" + 当前在分支上的位置（轻量）。

### 7.3 Review 体验（核心价值，属于中央内容区）

- 点改动文件 → 中央开 diff（复用编辑区与现有渲染）。
- 提供 **「逐项 review」模式**：上一处 / 下一处快速跳着看完所有改动，专治"AI 大改完的扫查"场景。
- 二进制 / 超大文件给占位提示，不硬渲染。

### 7.4 分支关系：轻量、按需、不常驻

定位里分支关系是**提交后确认用的**，不是天天盯的。因此：

- **绝不做全屏大图**，只展示最近一小段历史（当前分支 + 相关分支的最近 N 条），紧凑小图即可。
- 不占常驻空间，用完即走。

> 露出方式三选一，**待定**（后续确认）：
> - A. 分支名 popover：点分支名弹浮层画最近提交小图（最轻）。
> - B. 提交后临时滑出：提交成功后底部滑出紧凑历史小图展示"刚提交的 + 上下文"，看完收起（最贴流程）。
> - C. 右侧面板「变更 / 历史」两个 tab，历史 tab 放紧凑竖向小图（最省事，但窄栏里图较简）。

### 7.5 AI 入口

- 只做**一个唤起 codex / claude 的入口**（Raycast 式：一个唤起、给结果、用户确认）。
- AI 的产出丢进 diff，由用户 review 后提交。不做编排，不替用户做决定。

---

## 8. 一句话总结

> 用更少的重量，把"看清 AI 改了什么、顺手微调、优雅地提交"这件高频小事，做到原生、快、好看。
> 凡是会让它变重、变成编辑器或专业 Git 工具的功能，都不做——轻和优雅本身就是产品。
