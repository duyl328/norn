# 一、产品需求文档

## 2. 产品背景

在日常开发中，用户经常只是想快速查看代码、做少量编辑、查看 Git diff、提交代码或推送分支，但打开 IntelliJ IDEA、WebStorm、PyCharm、GoLand 等完整 IDE 会占用较多内存和启动时间。

本产品希望提供一个轻量级桌面应用，用于替代“为了看代码和提交 Git 而打开完整 IDE”的场景。

产品不追求成为完整 IDE，而是提供：

1. 快速打开项目目录
2. 轻量级代码编辑体验
3. 常见语言语法高亮
4. 常用编辑器快捷键，尽量贴近 JetBrains IDEA 系列
5. Git 状态查看、diff、暂存、提交、推送、拉取等核心操作
6. 较低资源占用和快速启动体验

------

## 3. 产品目标

### 3.1 核心目标

| 目标   | 说明                                           |
| ------ | ---------------------------------------------- |
| 轻量   | 相比完整 IDE，占用更少内存，启动更快           |
| 易用   | 操作体验接近 IDEA，降低 JetBrains 用户迁移成本 |

| 内置 Git 引擎   | 否       | 不使用 libgit2 / JGit，优先调用系统 Git CLI |

## 4. 目标用户

### 4.1 主要用户

1. 使用 JetBrains 系列 IDE 的开发者
2. 希望快速查看项目代码的开发者
3. 经常需要查看 diff、提交 Git，但不想打开完整 IDE 的用户
4. 多项目并行开发、需要减少内存占用的用户

### 4.2 典型使用场景

#### 场景一：快速查看项目

用户只想打开一个已有项目目录，查看几个文件，不希望启动完整 IDE。

#### 场景二：简单修改并提交

用户修改少量代码或配置文件，查看 Git diff，提交 commit 并 push。

#### 场景三：多项目同时处理

用户同时维护多个仓库，不想每个仓库都打开一个完整 IDE。

#### 场景四：代码审查前自查

用户在提交前查看 changed files、逐个检查 diff，确认无误后提交。

## 6. 总体架构建议

### 6.1 技术方向

推荐架构：

```text

本地 JSON / SQLite
```

### 6.2 架构原则

1. 编辑器核心保持轻量
2. Git 不内嵌核心实现，优先调用系统 `git`
3. 编辑器功能和 Git 功能解耦
4. 快捷键采用 Action System，不硬编码在 UI 事件里
5. Git 操作必须有明确错误反馈
6. 破坏性操作必须二次确认
7. 第一阶段不引入 LSP 和 Agent

# 五、Git 功能

## 17. Git 设计原则

### 17.1 底层实现

Git 功能第一阶段全部调用系统 `git` 命令。

不使用：

```text
libgit2
JGit
自研 Git 实现
```

### 17.2 为什么调用系统 Git

1. 使用用户已有 Git 配置
2. 支持 SSH key
3. 支持 credential helper
4. 支持 Git hooks
5. 支持 GPG signing
6. 支持 Git LFS
7. 行为与命令行一致
8. 降低实现复杂度

### 17.3 Git 命令调用方式

必须使用参数数组，不允许拼接 shell 字符串。

推荐：

```text
spawn("git", ["status", "--porcelain=v1"], { cwd: repoPath })
```

禁止：

```text
exec("git status --porcelain=v1 " + userInput)
```

原因：

1. 避免命令注入
2. 正确处理空格路径
3. 正确处理特殊字符
4. 跨平台更稳定

------

## 18. Git 仓库识别

### 18.1 识别规则

打开目录时：

1. 检查当前目录是否包含 `.git`
2. 如果没有，向父目录查找
3. 如果找到 Git root，则启用 Git 功能
4. 如果未找到，则 Git 面板显示“当前目录不是 Git 仓库”

### 18.2 命令

```bash
git rev-parse --show-toplevel
git rev-parse --abbrev-ref HEAD
```

### 18.3 验收标准

| 编号       | 标准                             |
| ---------- | -------------------------------- |
| AC-GIT-001 | 打开 Git 仓库后能显示当前分支    |
| AC-GIT-002 | 打开非 Git 目录时 Git 功能不可用 |
| AC-GIT-003 | 子目录打开时仍能识别 Git root    |

------

## 19. Git 状态

### 19.1 功能要求

Git 面板显示当前仓库变更文件列表。Norn 只提供轻量提交流程：用户勾选本次要提交的文件，提交时由后端按选中文件执行 `git add -A -- <files>` 和 `git commit`。不提供专业 Git 客户端式 staged / unstaged 双栏管理。

分类：

| 分类      | 说明           |
| --------- | -------------- |
| Modified  | 已修改文件 |
| Added     | 新增文件   |
| Deleted   | 删除文件   |
| Renamed   | 重命名文件 |
| Untracked | 未跟踪文件 |
| Conflict  | 冲突文件   |

### 19.2 推荐命令

```bash
git status --porcelain=v2 --branch -z
```

使用 `-z` 是为了更安全地处理文件名中的空格、换行和特殊字符。

### 19.3 UI 展示

每个文件展示：

1. 文件路径
2. 状态标记
3. 是否勾选为本次提交内容
4. 双击后打开 diff / 并排版本对比
5. 右键操作菜单

状态标记示例：

| 标记 | 含义      |
| ---- | --------- |
| M    | Modified  |
| A    | Added     |
| D    | Deleted   |
| R    | Renamed   |
| ??   | Untracked |
| !    | Conflict  |

## 20. Git Diff

### 20.1 Diff 类型

| 类型              | 说明       | 命令                        |
| ----------------- | ---------- | --------------------------- |
| 工作区对比 | 工作区文件相对 HEAD 的改动 | `git diff HEAD -- file` |
| 历史对比   | 某次提交里文件相对父提交的改动 | `git show` / `git diff` |

### 20.2 第一阶段展示方式

第一阶段支持统一 diff 解析和并排版本对比。暂不做 staged diff，因为当前产品不提供完整暂存区工作流。

### 20.3 Diff 面板要求

1. 显示文件路径
2. 显示新增行
3. 显示删除行
4. 显示上下文行
5. 支持从 diff 跳转到编辑器对应行
6. 大文件 diff 需要限制或提示

### 20.4 边界情况

需要处理：

| 情况         | 处理                          |
| ------------ | ----------------------------- |
| 新增文件     | 展示全部新增内容              |
| 删除文件     | 展示全部删除内容              |
| 二进制文件   | 显示“二进制文件无法预览 diff” |
| 大文件       | 提示用户确认加载              |
| 重命名文件   | 显示旧路径和新路径            |
| 文件权限变化 | 显示权限变更信息              |

------

## 21. 选择本次提交文件

### 21.1 第一阶段支持

第一阶段支持按文件或文件夹勾选本次要提交的内容。未勾选的文件不会被本次提交纳入。

### 21.2 操作

| 操作             | 说明                                      |
| ---------------- | ----------------------------------------- |
| 勾选文件         | 将文件纳入本次提交                         |
| 取消勾选文件     | 本次提交忽略该文件                         |
| 提交选中文件     | `git add -A -- <files>` 后执行 `git commit` |
| 提交并推送       | 提交成功后执行 `git push`                  |

### 21.3 UI 要求

1. 每个文件旁边有勾选框
2. 文件夹支持三态勾选
3. 提交框提示本次将提交的文件数量
4. 操作后自动刷新 Git 状态
5. 操作失败时显示错误信息

------

## 22. Commit

### 22.1 功能要求

用户可以在 Git 面板输入 commit message，并提交已勾选文件。

### 22.2 提交流程

```text
用户输入 commit message
  -> 检查 message 是否为空
  -> 检查是否至少勾选一个文件
  -> 执行 git commit
  -> 显示结果
  -> 刷新 Git 状态
```

### 22.3 推荐命令

```bash
git commit -m "message"
```

实现时应使用参数数组：

```text
git ["commit", "-m", message]
```

### 22.4 校验规则

| 校验             | 规则                            |
| ---------------- | ------------------------------- |
| 空 message       | 禁止提交                        |
| 未勾选文件       | 禁止提交                        |
| commit hook 失败 | 显示 hook 输出                  |
| GPG 签名失败     | 显示 Git 错误                   |
| 用户身份未配置   | 提示配置 user.name / user.email |

### 22.5 可选功能

第二阶段支持：

1. commit message 历史
2. amend commit
3. sign-off
4. 自动生成 message 模板
5. commit 后自动 push 选项

## 23. Push

### 23.1 功能要求

用户可以推送当前分支到远程仓库。

### 23.2 推荐命令

```bash
git push
```

如果当前分支没有 upstream，可以提示用户设置 upstream。

推荐命令：

```bash

```

### 23.3 需要处理的错误

| 错误               | 处理                 |
| ------------------ | -------------------- |
| 无远程仓库         | 提示未配置 remote    |
| 无 upstream        | 提示设置 upstream    |
| 权限失败           | 显示认证错误         |
| 被远程拒绝         | 提示需要 pull/rebase |
| 网络失败           | 显示网络错误         |
| pre-push hook 失败 | 显示 hook 输出       |

## 24. Pull

### 24.1 功能要求

用户可以从远程拉取当前分支更新。

### 24.2 推荐策略

第一阶段提供两种方式：

| 操作             | 命令                |
| ---------------- | ------------------- |
| Pull             | `git pull`          |
| Pull with Rebase | `git pull --rebase` |

默认策略可以读取用户 Git 配置，不强行覆盖用户习惯。

### 24.3 冲突处理

第一阶段不做可视化冲突解决。

当 pull 出现冲突时：

1. 显示冲突文件列表
2. 标记仓库处于冲突状态
3. 提示用户使用命令行或外部工具解决
4. 允许用户打开冲突文件手动编辑
5. 解决后允许用户继续提交

# 十六、风险与取舍

## 46. 主要风险

| 风险             | 说明                         | 应对                     |
| ---------------- | ---------------------------- | ------------------------ |
| 编辑器组件过重   | Monaco 可能偏重              | 优先评估 CodeMirror 6    |
| Git 输出解析复杂 | 文件名、编码、状态组合复杂   | 使用 `--porcelain -z`    |
| 跨平台差异       | Windows/macOS/Linux 路径不同 | 抽象 platform 层         |
| 快捷键冲突       | 系统快捷键和浏览器快捷键冲突 | 支持 keymap 配置         |
| 大仓库性能       | 文件树和 Git status 可能慢   | 忽略目录、增量刷新       |
| Git 冲突复杂     | conflict resolver 开发成本高 | 第一阶段只提示和打开文件 |
| 产品范围膨胀     | 容易变成完整 IDE             | 严格限制不做 IDE 功能    |

------

## 47. 关键取舍

### 47.1 Git CLI 优先

选择系统 `git` 而不是 libgit2 / JGit。

原因：

1. 实现成本低
2. 用户行为一致
3. 认证体系复用
4. hooks / LFS / GPG 支持更自然
5. 更符合轻量工具定位

### 47.2 CodeMirror 6 优先

如果产品目标是轻量，优先评估 CodeMirror 6。

原因：

1. 模块化
2. 可按需引入
3. 足够支持高亮、补全、折叠、搜索
4. 比 Monaco 更容易控制体积

如果后续需要更接近 VS Code 的完整编辑体验，再考虑 Monaco。

### 47.3 不做完整 IDE

本产品不是低配 IDEA，而是：

```text
轻量代码编辑器 + JetBrains 风格快捷键 + Git 管理工具
```

这是产品边界，不能被功能膨胀破坏。

------

# 十七、最终产品定义

本产品的核心价值是：

> 用更少资源完成“查看代码、简单编辑、检查 diff、提交 Git”的高频开发操作。

第一阶段应该优先确保：

1. 启动快
2. 打开项目快
3. 编辑稳定
4. Git 状态准确
5. Diff 清晰
6. Commit / Push 流程顺手
7. 快捷键接近 IDEA
8. 不被复杂 IDE 能力拖重

最终目标不是复刻 IDEA 或 VS Code，而是提供一个更轻、更专注的日常开发辅助工具。
